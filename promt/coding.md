### **阶段 1：项目原型生成 (开发前对齐)**

**目标：** 基于核心需求和初始用户流程，创建 AutoDevStudio 平台的交互式原型，以确保早期对齐并避免后续开发过程中的偏差。

**原型生成输入 (我将在提示词发布后单独提供这些内容，或假设您可访问此类平台的通用行业标准)：**

* **核心用户故事 (示例)：**

  * 作为用户，我希望能够登录并管理我的个人资料。
  * 作为用户，我希望能够通过提供需求（文本、文件、URL）来启动一个新项目。
  * 作为用户，我希望能够查看 AI 生成的需求分析（功能点、用户故事、非功能性需求）并提供反馈。
  * 作为用户，我希望能够查看 AI 推荐的技术栈并进行选择。
  * 作为用户，我希望能够审查 AI 生成的架构图和 UI 原型，并提供交互式反馈。
  * 作为用户，我希望能够监控 AI 驱动的开发进度（代码生成、测试）。
  * 作为用户，我希望能够与 AI 调试助手交互以解决问题。
  * 作为用户，我希望能够查看计费和使用情况信息。
* **关键设计原则：** 清晰、直观的用户界面，响应式设计，注重用户与 AI 的协作。
* **目标受众：** 软件开发者、产品经理、技术负责人。

**预期原型输出：**

1. **交互式 Web 原型：** 一个功能性、可点击的原型（例如，使用 Figma、Adobe XD 或基于 HTML/CSS/JS 的 Web 框架），演示以下主要用户流程：

   * 用户认证 (登录/注册)
   * 项目初始化 (需求输入)
   * 需求分析审查与反馈
   * 技术栈选择
   * 架构与 UI 原型审查及反馈循环
   * 开发仪表盘 (显示进度、测试、调试助手占位符)
   * 计费与使用概览
2. **设计理念总结：** 一份简要文档，概述关键设计选择、目标用户体验以及原型如何满足初始用户故事。
3. **线框图/模型图 (可选，如果作为交互式原型的补充)：** 关键屏幕的不同保真度级别。

**原型迭代与审批流程：**

* 在生成初始原型后，**请将其提交给我进行审查和反馈。**
* 我将提供迭代反馈，您需将其整合到后续原型版本中，直至获得批准。
* **至关重要的是，在我正式批准此原型阶段之前，不得开始全面开发。**

### **阶段 2：全面开发执行 (原型审批后)**

**目标：** 严格遵循我通过 GitHub 仓库提供的详细开发计划，系统地开发 AutoDevStudio 平台。

**开发计划访问：**

* 完整的开发计划，结构化为阶段、模块和详细任务（包括文件路径、核心函数定义、测试要求和完成检查清单），位于一个**GitHub 仓库中**。
* **您已被授予此仓库的访问权限，并应从 `main` 分支拉取最新版本的 `doc/development_plan.md` (或等效结构)。** 您将以此文档作为所有后续开发活动的主要指导。

**执行指南：**

1. **分阶段执行：** 严格按照开发计划分阶段、按任务执行。每个阶段（例如“阶段 6”）中的每个任务（例如“任务 010”）都必须按照其规范完成。
2. **AI Agent 集成：** 按照计划中详述的要求，实现指定的 AI Agent（RequirementAgent、CoderAgent、DebuggerAgent 等），确保它们与微服务架构的无缝集成。
3. **前端与后端开发：** 按照规范开发前端（React/TypeScript）和后端（Spring Boot/Python Agent Coordination Service）组件。
4. **测试驱动开发 (AI 驱动的端到端测试代码生成)：**

   * **主动生成 E2E 测试代码：** 在给定模块或功能集的核心系统编码完成后，并在最终“通过”之前，`TestAgent` (如**任务 021A** 所述) 必须**自主生成可执行的 E2E 测试脚本**（例如 Playwright/Cypress），基于 Gherkin 场景和 UI 原型。这些脚本应与应用程序代码一同进行版本控制。
   * **自动化执行：** 这些生成的 E2E 测试必须使用 `E2ETestRunner` (如**任务 026A** 所述) **自动执行**，针对已部署（或模拟部署）的应用程序版本。
   * **报告：** 必须生成详细的 E2E 测试报告（包括通过/失败状态、日志和失败截图），并使其可访问。
5. **性能和稳定性评估与优化：**

   * **自动化报告：** 一旦 E2E 测试通过，`PerfStabilityAgent` (如**任务 039** 所述) 必须**自主调用**，执行全面的性能和稳定性评估。这包括运行负载测试（如果需要，生成负载测试脚本）并根据定义的非功能性需求 (NFRs) 分析关键指标。
   * **主动优化循环：**

     * 如果性能和稳定性指标**未达到定义的 NFRs 或低于可接受的阈值**，`PerfStabilityAgent` 必须**主动识别瓶颈并触发优化动作。**
     * 此优化应涉及生成特定的 `OptimizationTask`，这些任务将由其他 Agent（例如 `CoderAgent` 用于代码级别优化，`RefactorAgent` 用于结构改进）执行。
     * 优化过程应该是**迭代和自动化的**，`PerfStabilityAgent` 在每轮优化后重新评估，直到满足 NFRs 或检测到无法解决的问题。
   * **报告：** 必须生成一份详细的性能和稳定性报告（可通过**任务 040** 的前端访问），其中概述发现、优化工作和最终状态。
6. **代码质量与安全性：** 保持高代码质量，遵循最佳实践，并集成计划中概述的安全措施（例如，代码沙箱、Prompt 过滤器、SecurityAgent）。
7. **版本控制 (GitHub)：**

   * 所有生成、修改或更新的代码都必须提交到指定的 GitHub 仓库。
   * 使用清晰、简洁的提交信息。
   * 为重要功能或阶段创建单独的分支，并提交 Pull Request (PR) 以供审查并合并到 `main` 分支。
   * 在 PR 描述中引用开发计划中相关的任务 ID 至关重要。

**沟通与进度报告：**

* 在每个主要阶段或逻辑任务组完成后，**请提供定期的进度更新。**
* **突出显示任何障碍、模糊之处或需要我输入的关键决策。**
* 在中间输出（例如，生成的代码模块、测试报告、架构图）可用时，提交它们以供审查。
